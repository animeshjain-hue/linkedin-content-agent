# What I Got Wrong

The most useful engineering documentation is the failures, not the architecture diagrams. These are the specific things that broke, why they broke, and what the fix actually was.

---

## 1. The token limit crash (Day 1 of automation)

The Ideator generates 5–10 ranked post angles, each with a `topic_lane`, `sub_topic`, `context_note`, `rationale`, `relevance_score`, `source_title`, and `source_url`. That is a lot of JSON for 10 angles.

I had set `max_tokens: 2000` as a global default in `config.yaml`. That is fine for the Writer (2 compact drafts at 150–300 words each sit comfortably under 2000 tokens of output). It is not fine for the Ideator.

What happened: the LLM would generate angles 1 through 7 or 8, then hit the token limit mid-JSON — typically truncating inside a string value. The `_parse_angles()` function in `src/agents/ideator.py` got a `JSONDecodeError` and crashed the whole run.

The failure mode was silent from the launchd perspective — the process exited with a non-zero code, nothing arrived on Telegram, and I had to grep `logs/launchd.log` to find it.

Fix: added `ideator_max_tokens: 4096` as a separate key in `config.yaml`. The Ideator reads it explicitly:

```python
max_tokens: int = config["agent_defaults"].get("ideator_max_tokens", 4096)
```

The global `max_tokens: 2000` remains for Writer and Critic. Lesson: per-agent token limits are not premature optimization — they are necessary because agents have very different output size profiles.

## 2. The topic diversity failure

The Ideator was generating angles that skewed heavily toward AI/agents content despite the content lane targets being 45% `ai_for_pms` and 35% `pm_craft`. Two bugs were compounding.

**Bug 1: the recent posts query used `status="posted"`.**

The intent was "show the model what I've posted recently so it avoids repeating themes." But at the time, only 1 post in the database had `status="posted"`. The other 46 seed posts were imported with `status="posted"` from the Excel export — but any drafts generated by the system sat at `status="draft"` or `status="approved"`. The model saw a near-empty recent history and had no avoidance signal.

**Bug 2: the prompt passed lane weights as context but never as a constraint.**

`topics.yaml` has a `lane_weight` field for each lane. The prompt mentioned the lanes but didn't tell the model to enforce a distribution. Left to its own judgment, the model naturally gravitated toward AI content because it's the easiest lane to generate a sharp PM angle for — every day brings new AI news with a clear "what does this mean for product teams?" hook.

Fix for Bug 1: changed to `exclude_statuses=["rejected"]` in `list_posts()`. This surfaces 15 recent posts regardless of whether they're drafts, approved, or posted — giving the model real theme signal from all non-rejected content.

```python
recent_posts = list_posts(
    settings.db_path,
    exclude_statuses=["rejected"],
    limit=15,
)
```

Fix for Bug 2: added `{lane_targets}` as a hard constraint variable in the ideator prompt, computed by `_compute_lane_targets()` in `src/agents/ideator.py`. It converts the `lane_weight` values into explicit angle counts: "ai_for_pms: ~5 of 10 angles (weight 45%)." The model treats this as a quota, not a suggestion.

Both fixes together went from "8 of 10 angles are AI topics" to a roughly lane-proportionate distribution.

## 3. The corporate SSL proxy

Every Anthropic API call failed on first run. The error was:

```
ssl.SSLCertVerificationError: [SSL: CERTIFICATE_VERIFY_FAILED]
```

The corporate network routes HTTPS through a proxy with a custom certificate authority. Python's `ssl` module uses its own certificate bundle and does not trust the corporate CA by default.

Fix: `truststore` library, which patches Python's SSL stack to use the system keychain (macOS Keychain, which already trusts the corporate CA). Two places this needs to happen:

1. In `scripts/run_daily.py`, before any network call:
   ```python
   import truststore
   truststore.inject_into_ssl()
   ```

2. In `src/outputs/typefully.py`, inside `schedule_post()`:
   ```python
   truststore.inject_into_ssl()
   ```

3. For CLI runs and uv dependency resolution:
   ```bash
   uv run --system-certs python scripts/run_daily.py
   ```

The `--system-certs` flag tells uv to use the system certificate store when making its own HTTPS calls (downloading packages, etc.). Without it, even `uv sync` can fail on the corporate network.

This is documented in `scripts/run_daily.py`'s module docstring:
```
uv run --system-certs python scripts/run_daily.py --idea "..."
```

## 4. The temperature deprecation surprise

Claude 4.x models silently ignore the `temperature` parameter — or more precisely, the API returns an error if you pass it. The Anthropic SDK changed the parameter handling between model generations, and the error message is not obvious.

The `call_llm()` wrapper in `src/llm.py` now detects the model name prefix and conditionally omits `temperature`:

```python
if not model.startswith("claude-opus-4") and not model.startswith("claude-sonnet-4") and not model.startswith("claude-haiku-4"):
    create_kwargs["temperature"] = temperature
```

This means the temperature values in `config.yaml` and in individual `call_llm()` calls are currently no-ops for all models in use. They're preserved for documentation and for forward compatibility if the model family changes.

## 5. What is probably wrong right now

**The Critic scoring is not validated against engagement.**

A Critic score of 8.5 might predict that a post performs well. It might not. The scoring rubric is calibrated against the voice guide and 47 seed posts, and it produces consistent scores (temperature 0.0 in `src/agents/critic.py`). But "consistent" and "predictive" are different things.

The known unknown: does the Critic's `hook_strength` score actually predict what makes someone stop scrolling? Does `voice_match` track with what gets positive comments? I don't know. The data to answer these questions doesn't exist yet.

This is why `auto_approve_threshold: 8.5` in `config.yaml` is currently unused, and why the Phase 4 auto-approve gate has an explicit data condition rather than a timeline: Critic scores correlate with engagement at r > 0.6 over a minimum of 20 posted samples. The correlation check is the Phase 3 Analyst agent's job. The threshold of 8.5 will be recalibrated when that data exists.

**The Ideator relevance scores have no ground truth.**

The `relevance_score` field on `RankedAngle` is generated by the Ideator model — it is an LLM judging which news items are relevant to my content lanes. This score is never validated against anything. The `update_relevance_score()` function in `src/inputs/news.py` exists to backfill scores from outside (e.g., from engagement data), but it has never been called. Until Phase 3, the Ideator's relevance ranking is entirely self-referential.
