# Data Model

All canonical schemas live in `src/schemas.py`. These are Pydantic v2 `BaseModel` subclasses. No raw dicts cross module boundaries — every piece of data that moves between modules is typed.

The SQLite schema is in `migrations/001_init.sql`. Schema changes require a new migration file. There is no ORM. Queries are plain SQL executed via `sqlite3.Connection`. The DB file is `data/brain.db`, gitignored.

These models are the most expensive things to change. Every migration is a production risk — the seed post corpus (47 posts, the foundation of the voice guide's format rankings) is already in the DB. Changing `Post` means migrating that data. Design decisions that touch these schemas deserve disproportionate scrutiny.

---

## Post

The central entity. Every draft generated by the WriterAgent, every seed post imported from the corpus, every approved and posted piece lives here.

```python
class Post(BaseModel):
    id: UUID
    created_at: datetime
    posted_at: datetime | None = None
    status: Literal["draft", "approved", "scheduled", "posted", "rejected"]
    body: str
    hook: str         # first 1-2 lines — the scroll-stopper, stored separately for display
    topic_lane: Literal["ai_for_pms", "pm_craft", "healthcare_ai"]
    sub_topic: str    # snake_case slug, e.g. "agentic_systems_for_pms"
    format: Literal["story", "framework", "contrarian", "list", "build_log", "question"]
    source_input_ids: list[UUID] = []   # voice notes / reading items that fed this post
    prompt_version: str                 # from YAML frontmatter of the prompt used
    model: str                          # exact model string, e.g. "claude-opus-4-7"
```

The SQLite column constraints enforce the same literals:

```sql
status     TEXT NOT NULL CHECK(status IN ('draft','approved','scheduled','posted','rejected'))
topic_lane TEXT NOT NULL CHECK(topic_lane IN ('ai_for_pms','pm_craft','healthcare_ai'))
format     TEXT NOT NULL CHECK(format IN ('story','framework','contrarian','list','build_log','question'))
```

**Status lifecycle:**

```
draft ──────────────────────────────────► rejected
  │
  ├─ (Telegram ✅ Approve) ──► approved ──► scheduled ──► posted
  │
  └─ (Telegram ✏️ Edit)    ──► approved ──► scheduled ──► posted
```

- `draft`: inserted by the orchestrator immediately after WriterAgent returns. Sent to Telegram.
- `approved`: set by the Telegram bot on ✅ or after edit. Triggers Typefully scheduling.
- `scheduled`: set when Typefully confirms the draft (currently handled as `approved` — the transition to `scheduled` is Phase 2 cleanup work).
- `posted`: set when Typefully reports the post went live (engagement ingestion, Phase 3).
- `rejected`: set by the Telegram bot on ❌. Excluded from IdeatorAgent theme-avoidance lists from this status onward.

**`hook` is stored separately** because it is displayed in:
- The Telegram message header
- The `format_posts_for_prompt()` reference blocks shown to the Writer
- The IdeatorAgent's theme-avoidance list
- The EvaluatorAgent's originality reference corpus (`seed_hooks`)

Storing it separately avoids extracting the first line from `body` everywhere.

**`prompt_version`** is critical for traceability. If a post approved six weeks from now has a quality problem, `prompt_version` tells you exactly which `prompts/writer.md` version generated it. This is why the YAML frontmatter `version` field is required, not optional.

**`source_input_ids`** is currently always `[]` — input tracking (voice notes, reading queue items, news item UUIDs) is Phase 4 work. The column exists now so the schema does not need to change when that work lands.

## EngagementSnapshot

Point-in-time capture of a post's LinkedIn metrics.

```python
class EngagementSnapshot(BaseModel):
    post_id: UUID
    captured_at: datetime
    impressions: int | None = None       # None until ingestion is wired
    reactions: int
    comments: int
    reposts: int
    profile_views_delta: int | None = None
```

`impressions` and `profile_views_delta` are nullable because LinkedIn does not expose them in all data access paths. The plan is to support multiple ingestion methods: Taplio API (paid, clean), manual LinkedIn analytics export, or scraping. The decision is deferred to Phase 3. The schema accommodates all three.

Multiple snapshots per post are expected — the same post can be snapshotted at 24h, 7 days, and 30 days. The AnalystAgent (Phase 3) will query this table to compute WoW engagement trends.

There is no PRIMARY KEY on `post_id` — the `id` column (a UUID, inserted via `uuid4()` in `posts.py::insert_engagement_snapshot`) is the PK. `post_id` is a foreign key to `posts(id)`.

## VoiceGuide

Loaded from `data/voice_guide.md` at Writer and Evaluator runtime. Not stored in SQLite — the markdown file is the source of truth, tracked in git.

```python
class VoiceGuide(BaseModel):
    sound_like: list[str]           # positive voice descriptors
    dont_sound_like: list[str]      # anti-patterns to avoid
    authority_topics: list[str]     # domains where the author has standing
    example_sentences: list[str]    # verbatim sentences from real posts
    forbidden_phrases: list[str]    # "delve", "tapestry", "game-changer", etc.
```

The `forbidden_phrases` list is enforced in two places: the Writer prompt (do not use these) and the Critic/Evaluator scoring rubric (deduct hygiene points if found). This double enforcement is intentional — the Writer can still produce a forbidden phrase if the model ignores the instruction; the Critic will catch it.

`voice_guide.md` was built from a full analysis of the 47-post corpus. It is the most valuable document in the system after the DB itself. Changes to it require re-running the voice-match eval suite.

## NewsItem

One row per unique URL fetched from RSS. Deduplication is by URL — the `id` is a deterministic UUID computed as `UUID(hashlib.md5(url.encode()).hexdigest())`. The same article fetched on multiple runs produces the same UUID, so `INSERT OR IGNORE` handles dedup without any separate state.

```python
class NewsItem(BaseModel):
    id: UUID              # deterministic from URL hash — dedup key
    fetched_at: datetime
    source: str           # feed name: "et_tech", "yourstory", "inc42", etc.
    url: str              # UNIQUE in SQLite
    title: str
    summary: str          # first 500 chars of RSS description tag
    relevance_score: float | None = None   # None until scored by Ideator (future)
    used_in_post_ids: list[UUID] = []      # backfilled when a post cites this item
```

`relevance_score` is currently always `None` in the DB. The Ideator scores angles in its own context window but does not write scores back to `news_items`. The intention (Phase 3) is to backfill scores so the Analyst can understand which sources consistently produce high-relevance angles. The column exists now to avoid a schema migration later.

`used_in_post_ids` is likewise currently always `[]`. Backfilling this when an approved post cites a news item requires linking `RankedAngle.source_url` through to the `NewsItem.url` — this is wiring that will happen in Phase 3 when full provenance tracking becomes useful for the Analyst.

The SQLite column is `url TEXT NOT NULL UNIQUE`, making the DB itself the dedup enforcer — even if `_url_to_uuid()` were to produce a collision (it won't for different URLs), the UNIQUE constraint on `url` would prevent a second insert.

## StrategyDoc

Generated weekly by the StrategistAgent. Written to `data/strategy_current.md`, gitignored and regenerated each week. The Pydantic model defines the schema; the file format is implementation detail.

```python
class StrategyDoc(BaseModel):
    week_of: date
    lane_mix: dict[str, float]        # {"ai_for_pms": 0.6, "pm_craft": 0.35, "healthcare_ai": 0.05}
    target_frequency: int             # posts per week
    preferred_formats: list[str]      # formats to lean into this week
    preferred_post_times: list[str]   # e.g. ["Tuesday 8am", "Thursday 7am"]
    experiments: list[str]            # deliberate format/hook trials to run
    avoid: list[str]                  # formats or topics that underperformed last week
    rationale: str                    # data-backed explanation for this week's decisions
```

`lane_mix` values are floats that should sum to 1.0 (not enforced at the Pydantic level — the StrategistAgent prompt is responsible for this).

`experiments` is the field that gives the system the ability to learn. An example entry: `"try one contrarian hook mid-week — last week's contrarian post outperformed the framework posts 3x on comments"`. These are deliberate, scoped changes with an implicit measurement plan.

`avoid` contains explicit exclusions derived from data: `"no list-heavy posts — last week's two list posts averaged 2.1K impressions vs 8.4K for the strategy teardown"`. This is how engagement data feeds back into content strategy without requiring Animesh to manually synthesise it.

Not used until Phase 3. The schema is defined now so the Ideator can reference `strategy_current.md` without a schema migration when Phase 3 lands.

## DB access patterns

All DB operations go through `src/brain/db.py::get_connection()`, which returns a `sqlite3.Connection` with `row_factory = sqlite3.Row`. This allows column access by name (`row["status"]`) rather than index.

Every function in `src/brain/posts.py` opens and closes its own connection via context manager. No persistent connection pool — SQLite on a local file does not need one, and a persistent connection would complicate the async Telegram bot context.

The `schema_migrations` table tracks applied migrations by name. New migrations are `.sql` files in `migrations/`. Running them is currently manual — `sqlite3 data/brain.db < migrations/002_next.sql`. Migration tooling will be added in Phase 3 when the schema is more stable.
